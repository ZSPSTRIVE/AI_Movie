# Python RAG 面试速记

这份文档只保留最常见、最容易被追问的主流面试点，目标是方便快速学习和找实习时稳定输出。

## 1. 30 秒项目介绍

这是一个极简版电影 RAG 服务，外部仍然保留上传和搜索能力，但内部只做 4 件事：

1. 把知识库文档或电影信息切分后入库
2. 优先走 Milvus 向量检索
3. Milvus 或 embedding 不可用时退化到 PostgreSQL 关键词检索
4. 对电影类查询，先调用 `tvbox-proxy` 拉取影片信息，再增量写回本地知识库

一句话概括：`这是一个“向量检索优先、数据库兜底、搜索时可增量补库”的最小 RAG 项目。`

## 2. 为什么这样设计

### 为什么保留 PostgreSQL

- 存 `rag_document` 和 `rag_chunk` 元数据
- 方便回表展示引用来源
- Milvus 挂掉时还能做关键词兜底检索

### 为什么保留 Milvus

- 向量检索比纯 `LIKE` 更适合语义问答
- 但它是增强层，不是单点依赖

### 为什么不做 BM25、reranker、query enhancement

- 这些会增加配置项、失败路径和解释成本
- 对实习项目来说，“一条最小主链跑通”比“很多增强模块堆在一起”更重要
- 面试时更容易把链路讲清楚

## 3. 你要能讲清楚的链路

### 文档入库链路

`/rag/ingest`

1. 读取文本
2. 切分 chunk
3. chunk 写 PostgreSQL
4. 如果 embedding/Milvus 正常，再写向量库
5. 清理搜索缓存

### 搜索链路

`/rag/search`

1. 如果像电影查询，先调用 `tvbox-proxy` 补充本地知识库
2. 优先向量检索
3. 向量失败或无命中时，走 PostgreSQL 关键词检索
4. 拼上下文，调用 LLM 生成答案
5. 返回 answer + references

## 4. 最常见面试题

### 1. 为什么要切分 chunk

原始文档太长，直接送给模型成本高、噪声大。切分后更容易定位最相关片段，也方便做引用回溯。

### 2. 为什么要把 chunk 元数据单独放 PostgreSQL

向量库更适合做相似度检索，不适合承担完整业务元数据管理。把文档标题、来源、chunk 文本放 PostgreSQL，方便更新、删除、回表和引用展示。

### 3. 为什么还要做 Redis 缓存

搜索请求会重复，特别是相同 query。缓存能减少重复检索和模型调用，降低响应时间和成本。

### 4. 为什么要做 PG 兜底检索

因为在真实项目里，向量模型、远端 embedding API、Milvus 都可能失败。PG 兜底可以保证最小可用性，属于典型的降级设计。

### 5. 电影搜索为什么要先调用 TVBox API

本地知识库不可能一开始就覆盖全部电影。搜索时按需拉取外部 API，再把结果写回本地，这样能做到“越查越全”，同时避免一次性全量抓取。

### 6. 为什么首页推荐不能随机

随机刷新会导致用户体验不稳定，也不利于后台运营控制。推荐位应该由后台在某个时间点生成和发布，用户在一个周期内看到的是稳定结果。

### 7. 如果 Milvus 不可用怎么办

当前实现会直接退化到 PostgreSQL 关键词检索，保证服务还能返回结果。也就是说：`Milvus 是增强，不是单点。`

### 8. 为什么电影知识同步要带 `replace_by_source`

同一部电影重复同步时，如果不按 `source_path` 替换，旧版本 chunk 会残留，导致重复命中。`replace_by_source` 的作用就是保持“一部电影只有一份当前有效知识”。

### 9. 引用来源为什么重要

RAG 和普通聊天最大的区别就是“答案要有依据”。返回 references 能让用户知道答案是从哪个文档、哪个 chunk 来的，也方便排错。

### 10. 这个项目最大的取舍是什么

为了降低学习成本，主动放弃了很多增强模块，只保留主链。这样做的代价是上限没那么高，但换来更低的复杂度和更强的可解释性。

## 5. 腾讯实习面试时推荐说法

可以这样说：

`我做的是一个简化版 RAG 服务，重点不在堆模块，而在把检索、入库、降级、缓存、引用这条链路做扎实。向量检索正常时走 Milvus，异常时退化到 PostgreSQL 关键词检索；电影类查询还能按需调用外部 API 增量补库。这样既有工程完整性，也方便我清楚解释每个设计取舍。`

## 6. 不要硬讲的内容

下面这些如果没被问到，不建议主动展开：

- reranker 原理细节
- BM25 数学公式
- LangChain 各种高级 chain/agent
- 多知识库路由
- 复杂的指标系统和熔断系统

因为当前项目刻意没有走那条高复杂度路线。
